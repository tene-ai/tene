## 리소스별 병목점 맵

| 계층 | 리소스 | 스펙 | 한계 |
|------|--------|------|------|
| **L1. ALB** | Application LB, 2 AZ | 자동 확장 | ~10,000 req/sec (사실상 무제한) |
| **L2. ECS** | 0.25 vCPU / 512 MiB x 2 tasks | 최대 3 tasks (CPU 70% 기준 자동 확장) | 태스크당 ~200-400 req/sec |
| **L3. RDS** | db.t4g.micro (2 vCPU, 1 GB RAM) | 최대 ~85 커넥션 | ~300-500 queries/sec (단순 CRUD) |
| **L4. NAT** | fck-nat t4g.nano | 5 Gbps burst | 외부 호출 시에만 사용 (OAuth, LemonSqueezy) |
| **L5. S3** | Standard | 3,500 PUT/sec, 5,500 GET/sec | vault push/pull 전용, 병목 아님 |
| **L6. Rate Limit** | 유저별 토큰 버킷 | Free: 100 RPM, Pro: 1,000 RPM | 앱 레벨 스로틀링 |

## 동시 요청 처리량

실질적 병목은 **RDS db.t4g.micro** (최대 ~85 커넥션).

```
ECS 2 tasks (현재 정상 상태)
  -> 실제 사용 가능 DB 커넥션: ~40-60
  -> 단순 쿼리 기준: ~300-500 req/sec

ECS 3 tasks (자동 확장 최대치)
  -> 실제 사용 가능 DB 커넥션: ~60-80
  -> ~450-700 req/sec (RDS가 먼저 포화)
```

| 시나리오 | 동시 접속 | 초당 요청 | 상태 |
|----------|:---------:|:---------:|:----:|
| 가벼운 사용 (list/get) | ~200 | ~300 req/sec | 여유 |
| 보통 사용 (push/pull 혼합) | ~100 | ~150 req/sec | 여유 |
| 무거운 사용 (push 집중) | ~50 | ~50 req/sec | 여유 (S3 IO 의존) |
| **피크 한계** | **~300-500** | **~500 req/sec** | RDS 포화 시점 |

## 사용자 수용량 추정

### 사용 패턴

Tene은 시크릿 관리 CLI 도구로, 상시 접속 서비스가 아닙니다. 일반적인 개발자 사용 패턴:

- `tene push/pull`: 개발 중 하루 수 회
- `tene run`: 서버 시작 시 1회 (시크릿 주입)
- `tene set/get`: 간헐적
- Dashboard: 가끔 확인

**일반적인 개발자 API 호출량**: 하루 ~50-100회, 피크 시 분당 ~5회

| 사용자 유형 | 피크 RPM | Rate Limit |
|------------|:--------:|:----------:|
| Free 사용자 | ~5 req/min | 100 RPM |
| Pro 사용자 | ~10 req/min | 1,000 RPM |

### 수용 가능 사용자 규모

| 지표 | 수치 | 근거 |
|------|:----:|------|
| **동시 활성 사용자** | **200-300명** | RDS 85 conn 한계, ECS 2-3 tasks |
| **일일 활성 사용자 (DAU)** | **1,000-2,000명** | 동시접속률 ~15-20% 가정 |
| **전체 가입자** | **5,000-10,000명** | DAU/MAU 비율 ~20-30% |
| **Pro 유료 사용자** | **200-500명** | Pro RPM 1,000으로도 현재 인프라 충분 |

| 단계 | 사용자 규모 | 필요 조치 | 추가 비용 |
|------|:-----------:|-----------|:---------:|

### 첫 번째 스케일링 포인트

가장 먼저 한계에 도달하는 리소스는 **RDS db.t4g.micro** (1 GB RAM, ~85 최대 커넥션).

업그레이드 경로: db.t4g.micro -> db.t4g.small (월 +$12, RAM 2배 2 GB, ~170 커넥션)

이 단일 변경으로 수용량이 ~2,000 DAU에서 ~5,000 DAU로 확장됩니다.

## 주요 분석 결과

1. **CLI 기반 사용 패턴이 유리하게 작용.** 상시 접속 웹앱과 달리 Tene 사용자는 간헐적으로 API를 호출합니다. 동일 인프라에서 일반 웹앱 대비 **3-5배 더 많은 사용자**를 수용할 수 있습니다.

2. **첫 번째 병목은 ECS가 아니라 RDS.** ECS Fargate는 최대 3 tasks까지 자동 확장되지만, 모든 태스크가 동일한 RDS 인스턴스를 공유합니다 (~85 커넥션 한계). ECS를 3 이상으로 늘리려면 RDS를 먼저 업그레이드해야 합니다.

3. **S3와 ALB는 현 규모에서 사실상 무제한.** S3는 3,500+ PUT/sec을 처리하고, ALB는 자동으로 투명하게 확장됩니다.

4. **Rate Limiting이 보호 역할.** Free 100 RPM, Pro 1,000 RPM 제한으로 단일 사용자가 공유 리소스를 고갈시키는 것을 방지합니다.

5. **현재 인프라 비용 효율성.** ~$117/월 (Multi-AZ 포함) 기준, 최대 2,000 DAU 수용 시 사용자당 비용은 ~$0.06/월로 초기 단계 제품에 매우 효율적입니다.
