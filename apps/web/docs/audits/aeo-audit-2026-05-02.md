### A.1 — `src/middleware.ts` (NEW, untracked)
Severity: P0

Status: 🔵 WIP-on-this-branch — works as written, but has critical scope issues.
Evidence: `src/middleware.ts:13-23` — `ALLOWED_HOSTS = new Set(["tene.sh", "www.tene.sh"])`; matcher `["/((?!_next|api|.*\\.[^/]+$).*)"]`.
Verified live: `curl -I http://localhost:3001/cli` returned `x-robots-tag: noindex, nofollow` (correct on dev host).
Why it matters: Prevents Vercel preview URLs (e.g. `tene-{hash}-agent-kay-project.vercel.app`) from being indexed in place of `tene.sh`. The matcher correctly excludes `_next` static and `/api` and any extension-bearing path.
Remediation:
1. The matcher excludes `.*\.[^/]+$` (anything with a dot) — which means `/llms.txt`, `/llms-full.txt`, `/install.sh`, `/sitemap.xml`, `/robots.txt`, `/og-image.webp` all bypass middleware. On a non-allow-listed host (e.g. preview), these files would still be served WITHOUT the noindex header. Risk: a preview's `llms.txt` could leak into AI training. Fix: change the matcher exclusion list to be allowlist-style instead of extension-blacklist, or special-case `.html` only.
2. `www.tene.sh` is in the allow list but the canonical is `tene.sh` (no www). Without a 301 from www → apex, both hosts respond 200 and Google may still pick the wrong canonical. Either drop `www.tene.sh` from `ALLOWED_HOSTS` and let it 404/redirect via DNS+Vercel, OR add a true 301 redirect in `next.config.ts`.
3. `host` header strip lowercases but does not handle X-Forwarded-Host (Vercel sets `host`, but proxies sometimes set X-Forwarded-Host; Next 16 normalizes this internally — verify in Vercel runtime by deploying a preview branch and `curl -I` it).
Acceptance criterion: `curl -sI https://tene-{hash}.vercel.app/blog/ai-reads-env | grep -i 'x-robots-tag'` returns `noindex, nofollow`. Same probe against `https://tene.sh/blog/ai-reads-env` returns no x-robots-tag.
